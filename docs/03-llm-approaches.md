# 3. LLMエージェント手法

暗号誤用検出にLLMを適用する研究は急速に進展している。

## 3.1 CryptoScope LLM

- **論文**: arXiv:2508.11599
- **手法**: Chain-of-Thought (CoT) + Retrieval-Augmented Generation (RAG)
- **知識ベース**: 12,000件の暗号知識エントリ
- **性能**: GPT-4o-miniベースラインから20%改善
- **成果**: 未知の暗号脆弱性9件を実世界のコードベースから発見
- **ベンチマーク**: OWASP + CryptoAPI-Bench で評価

### 本プロジェクトとの関係
CryptoScope LLM はソースコードを入力とする **ホワイトボックス** 手法。本プロジェクトは同様のLLMエージェント技術を **ブラックボックス観測** に適用する点で新規性がある。

## 3.2 Beyond Static Tools

- **論文**: arXiv:2411.09772
- **主張**: LLM（GPT-4等）が既存の静的解析ツール（CryptoGuard、CogniCrypt、Snyk Code）を上回る検出性能を示す
- **ベンチマーク**: OWASP、CryptoAPI-Bench、MASC
- **結果**: プロンプトエンジニアリング適用後、F-measure 94.6%。10カテゴリで既存SOTAを超過
- **検証**: CamBenchでも汎化性能を確認

### 示唆
LLMの暗号知識は十分に豊富であり、ツール実行結果の「解釈」にLLMを活用するアプローチは有効。本プロジェクトではLLMを「ツール実行の判断者」として使い、解釈と仮説更新を担わせる。

## 3.3 KG+LLM フレームワーク

- **論文**: arXiv:2601.03504
- **手法**: ナレッジグラフ + LLM + Human-in-the-Loop (HIL)
- **目的**: PQCレディネススコアリング
- **特徴**: 組織の暗号資産をグラフ構造で表現し、LLMが移行アドバイスを生成

### 本プロジェクトとの関係
出力フォーマット（CBOM）とPQC脆弱性評価の設計に影響。ナレッジグラフ的な暗号知識の構造化は将来的に取り込める。

## 3.4 LLM手法の共通課題

| 課題 | 影響 | 本プロジェクトでの対策 |
|------|------|---------------------|
| 幻覚（Hallucination） | 存在しない脆弱性を報告 | 確信度スコア + 複数チャネル裏付け |
| コンテキスト長制限 | 大規模スキャン結果の処理 | フェーズ分割 + 要約的入力 |
| 非決定性 | 同一入力で異なる結果 | 温度0.1 + 反復検証 |
| コスト | APIコール数 | 最大反復回数制限 + 収束条件 |
| 再現性 | 結果の再現が困難 | ツール実行結果をログ保存 |
